**Known Issues & Architectural Reflections:**
During testing, I observed some flaws in the LLM orchestration loop. I chose to build the LLM integration from scratch using the raw Google GenAI SDK to deeply understand the mechanics. However, this exposed a few issues that an orchestration framework (like LangChain) typically handles out-of-the-box:

1.  **LLM Schema Hallucination (Text-to-SQL):** When using the `sql.query` tool, the LLM often assumes a flat table structure (e.g., trying to query `customers` instead of the dbt-transformed `marts.dim_customers`). While the LLM attempts to self-correct by querying `information_schema.tables`, it wastes tokens and turns. *Reflection: If I had used LangChain's SQL Toolkit, the framework would have automatically injected the database schema into the context, preventing these hallucinations entirely.*
2.  **Embedding Model Error Handling:** The `kb.search` tool occasionally encounters a `404 Not Found` error when calling `models/text-embedding-004`. The backend currently catches this and generates dummy vectors as a fallback to prevent a crash, which degrades search quality. *Reflection: Relying on raw SDK exception handling without a robust retry/circuit-breaker mechanism (which frameworks often provide) leads to poor "fail fast" behavior.*
3.  **Max Turn Limit Exceeded:** Because of the two issues above, the raw loop logic sometimes hits the maximum turn limit (5 turns) while the LLM struggles to correct itself, resulting in the fallback response.

Building this from scratch was a great learning experience regarding the complexities of Agentic workflows, and I now clearly see the value of adopting standardized frameworks for Context Provisioning (RAG/SQL) in production.
